# Architecture

## Repository structure

The repository follows a two-level layout: a `modules/` directory with reusable domain modules, and a `projects/` directory where each client has its own configuration that instantiates those modules.

```
nexly-infra/
├── modules/
│   ├── lambda-etl/          # Lambda function + IAM role + log group + alarms
│   ├── lambda-container/    # Lambda via container image (ECR)
│   ├── eventbridge-schedule/ # Cron rule + Lambda target + invoke permission
│   ├── cloudwatch-alarm/    # Metric alarm + SNS action
│   └── secrets-manager/     # Secret resource (value set out-of-band)
└── projects/
    └── silva-nutrition/
        ├── main.tf          # Module instantiation
        ├── observability.tf # Alarms + dashboard
        ├── backend.tf       # S3 + DynamoDB config
        ├── providers.tf
        ├── variables.tf
        ├── outputs.tf
        └── lambdas/         # Python source code per function
```

The design principle is that adding a new client means copying `projects/silva-nutrition/`, updating `terraform.tfvars`, and running `terraform apply`. No module code changes, no copy-pasting resource blocks.

## Module design: lambda-etl

Each ETL Lambda gets a dedicated module instantiation that creates the function, its IAM role with least-privilege policies, its CloudWatch log group, and its standard alarms (error rate and duration). This means a new Lambda is never deployed without monitoring.

```hcl
module "sync_clientes" {
  source = "../../modules/lambda-etl"

  function_name     = "silva-nutrition-sync-clientes"
  runtime           = "python3.12"
  handler           = "handler.lambda_handler"
  timeout           = 300
  memory_size       = 512
  source_code_path  = "${path.module}/lambdas/sync_clientes"

  environment_variables = {
    SUPABASE_SECRET_ARN    = data.aws_secretsmanager_secret.supabase.arn
    WOOCOMMERCE_SECRET_ARN = data.aws_secretsmanager_secret.woocommerce.arn
    ENVIRONMENT            = "production"
  }

  layer_arns         = [aws_lambda_layer_version.shared.arn]
  sns_alarm_topic_arn = aws_sns_topic.alerts.arn
}
```

## State management

Remote state is stored in S3 with versioning enabled, and DynamoDB handles state locking to prevent concurrent `terraform apply` executions from corrupting the state file. The locking mechanism became important during CI/CD setup, where PR and main branch pipelines could theoretically run simultaneously.

```hcl
terraform {
  backend "s3" {
    bucket         = "nexly-terraform-state"
    key            = "projects/silva-nutrition/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "nexly-terraform-locks"
    encrypt        = true
  }
}
```

## CI/CD pipeline

Two GitHub Actions workflows handle the full lifecycle.

**terraform-ci.yml** runs on every PR and executes `fmt -check`, `init`, `validate`, and `plan`. The plan output is posted as a comment on the PR so reviewers can see exactly what will change before approving. This catches configuration errors and unintended resource replacements before they reach production.

**terraform-cd.yml** runs on merge to main. It regenerates the plan and then waits for a manual approval in the GitHub `production` environment before running `apply`. The environment requires at least one reviewer from the infrastructure team. The approval window is 24 hours before the workflow expires.

This setup means no change reaches production without a written record of what was planned, who approved it, and when it was applied.

## Lambda deployment strategy

Eight of nine Lambdas use Zip deployment with a shared layer. One Lambda, `recalc_clusters`, uses a container image via ECR. The reason is that scikit-learn and numpy together exceed the 250MB unzipped limit for Zip packages. A container image removes that constraint without requiring any changes to the module interface.

```
recalc_clusters container: ~380MB (scikit-learn + numpy + handler)
All other Lambdas: ~5MB each via Zip + shared layer
```

## IAM: least privilege per function

Each Lambda has its own IAM role. The role policy is generated by the module based on the environment variables passed in: if a function receives a `SUPABASE_SECRET_ARN`, the module automatically adds `secretsmanager:GetSecretValue` for that specific ARN. Nothing broader. No function can read secrets it was not explicitly configured to access.

## Shared Lambda layer

A Python Lambda Layer carries the seven shared modules used across all ETL functions.

```
layers/shared/python/lib/
├── bling_client.py      # OAuth2 refresh with distributed lock
├── config.py            # Secrets Manager client with TTL cache
├── dynamodb_lock.py     # Distributed lock implementation
├── logger.py            # Structured JSON logging
├── metrics.py           # CloudWatch PutMetricData wrapper
├── supabase_client.py   # PostgreSQL async client
└── woocommerce_client.py # REST API v3 with pagination and rate limiting
```

This layer is versioned: a new version is published and all functions are updated together via `terraform apply`.
